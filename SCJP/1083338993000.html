[SCJP Tips] [SCJP Tips: Operators and Assignments] [SCJP Tips: Threads] Thread Synchronization

Once again, I'm going to start right off with a mock exam question I made up. Take a look at this and see what you come up with for an answer.

What is the result of compiling and running the following code? Select the one correct answer. 
class Q010 implements Runnable
{
    String semaphore = new String("");
    String name = new String("");
    
    public Q010(String n)
    {
        name = n;
        Q010a.getLock(semaphore);
    }
    
    public void run()
    {
        synchronized(semaphore)
        {
            for ( int i = 0; i < 10; i++ )
            {
                System.out.println("Message from " + name + ".");
            }
        }   
    }
}

public class Q010a
{
    private static String s = new String("g");
    
    public static void main(String[] args)
    {
        Q010 q1 = new Q010("q1");
        Q010 q2 = new Q010("q2");
        
        Thread t1 = new Thread(q1);
        Thread t2 = new Thread(q2);
        
        t1.start();
        t2.start();
    }
    
    static void getLock(Object o)
    {
        o = s;
    }
}

A. Compilation Error

B. The output consists of 10 lines of "Message from q1" followed by 10 lines of "Message from q2".

C. The output consists of 10 lines of "Message from q2" followed by 10 lines of "Message from q1".

D. The output consists of 10 lines of "Message from q2" and 10 lines of "Message from q1" but we can not determine which block will appear first.

E. We can not determine the output.

So what did you mark as an answer?

First of all, this code, although not especially pretty, compiles fine, so A is not the correct choice.

If you selected B or C, you'd be wrong, as well. It's up to the underlying operating system to actually start a thread so we have no idea which thread, t1 or t2, will actually enter the running state first. Therefore, we can't be sure which line will be output first.

And if you answered D, you'd be close, but still wrong. The answer is E - we can not determine what the output will be prior to execution.

Why can't we determine the output? Well, the culprit is our synchronization - it's faulty. Let's start by looking at the constructor of class Q010. 

public Q010(String n)
{
    name = n;
    Q010a.getLock(semaphore);
}

...

private static String s = new String("g");

...

static void getLock(Object o)
{
    o = s;
}

This whole problem really boils down to one thing - objects are passed by passing a copy of a reference to that object. That means that, when we pass an object to a method, we don't actually pass the object, we pass a reference to that object. In this case, we're passing the reference variable semaphore to the method getLock(Object). What does that give us? Well, that gives us two references to the same object. The reference variable semaphore, in class Q010, and the formal parameter o, in Q010a.getLock(Object), both reference the same object. In this case, that object is an empty String - it was initialized as a instance member initializer.

So what does the getLock(Object) method do? Well, in short, just about nothing. It assigns a reference to Q010a.s, which is a String object containing the String "g", to the reference variable o. But, keep in mind that o is just a reference to an object, not the object itself. By assigning a new reference value to o, we have not changed the original object or the original reference variable, semaphore. The variable semaphore still happily references the empty String object it always referenced.

Because I love pictures so much (especially when dealing with references, here's a before and after picture with regards to that line in Q010a.getLock(Object):

Before: 
               +-----+
semaphore ---> | ""  | <--- o
               +-----+

               +-----+
Q010a.s -----> | "g" |
               +-----+
After: 
               +-----+
semaphore ---> | ""  |
               +-----+

               +-----+
Q010a.s -----> | "g" | <--- o
               +-----+

Therefore, we end up with two Runnable objects that each are synchronizing on their own, separate, locks. Because each Runnable object is using its own object to synchronize on, each thread will always have an available lock and the two threads will not synchronize with one another. Therefore, we have no idea what the output of this program will be. In fact, here's the output I got: 

Message from q1.
Message from q1.
Message from q1.
Message from q1.
Message from q1.
Message from q1.
Message from q1.
Message from q1.
Message from q2.
Message from q2.
Message from q2.
Message from q2.
Message from q2.
Message from q2.
Message from q2.
Message from q2.
Message from q2.
Message from q2.
Message from q1.
Message from q1.

I hope you got that question right and, even if you didn't, I hope you understand the answer now. The type of trickery I used here isn't uncommon for the SCJP exam. The author of the question wants to lead you one way, just as I wanted to make you think this question was about threads and synchronization, and then throw a curveball that's unrelated, such as how objects are passed to methods. That's one of the reasons it's important to read every question carefully. In all my time studying for the SCJP, the most difficult thing for me to do was to take my time to ensure that I didn't miss a little detail.

Hopefully, you learned something today, even if it was just to read SCJP questions very carefully. Have a good weekend, everyone.

Comment from Jeroen Wenting on 03 May 2004 01:32:58 CDT
Reply | Permalink Re: Thread Synchronization
I think everyone here understands how objects are passed around in Java - I think there's just some confusion about what constitutes pass by value and pass by value.

Pass by Value means that we make a copy of the variable we are passing. Pass by Reference means we do not. Obviously, this is confusing because we do make a copy of the reference variable, but we do not make a copy of the object it refers to. So which is it? Pass by reference or pass by value?

Well, in the most technical terms, we are passing a variable, which is a reference variable, to a method by copying the value of that variable to the activation stack. That, essentially, is pass by value. Of course, the variable we are passing is a reference, but it doesn't change the fact that we just made a copy of that variable. Because we make that copy, I'd say that Java is a "pass by value" language.

There's a neat article you can check out here that goes into some greater detail, as well.
